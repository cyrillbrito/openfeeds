# Database Migrations

Drizzle SQL migrations, applied by a one-shot service before app services start. Forward-only.

## Shape

- SQL files: `packages/db/drizzle/` (generated by `drizzle-kit`).
- `runMigrations()` in `packages/db/src/migrate.ts` applies pending migrations.
- `apps/migrator` calls it and exits 0/1.
- `docker-compose.yml`: `migrator` runs with `restart: 'no'`; `web`/`worker` gate on `service_completed_successfully`.

## Why a dedicated service (not on api startup)

- **Single writer.** N replicas booting won't race `migrate()`; no cold-start blocked behind the slowest migration.
- **Fail-closed.** Bad migration → migrator exits 1 → new app version never starts → old version keeps serving.
- **Separation.** App images stay runtime-only; no SQL files or `drizzle-kit` in web/worker.
- **Independently runnable.** Apply migrations without deploying app code (e.g. `CREATE INDEX CONCURRENTLY`).

## Forward-only (expand/contract)

1. **Expand** — additive migration. Ship.
2. **Migrate code** — read/write new shape. Ship.
3. **Contract** — drop old column/table. Ship.

Recovery from a bad schema change is a new forward migration.

## Version pinning

`docker-compose.yml` pins all three images to `${OPENFEEDS_VERSION:-latest}` so a deploy moves migrator and app in lockstep.

## Running

- **Local:** `bun migrate`. App processes don't migrate.
- **CI:** `bun --cwd=apps/migrator src/index.ts` before tests.
- **Prod:** `docker compose up` — migrator runs, then web/worker start.

## Not in scope

Long-running data backfills needing domain code belong in BullMQ jobs, not `apps/migrator`.
