# Database Migrations

Drizzle SQL migrations. Forward-only. Applied automatically on server boot.

## Shape

- SQL files: `packages/db/drizzle/` (generated by `drizzle-kit`).
- `runMigrations()` in `packages/db/src/migrate.ts` applies pending migrations.
- `apps/server/src/index.ts` calls `await runMigrations()` before Hono starts listening. A failed migration throws → process exits non-zero → deploy fails loudly.

## Why on server boot (not a dedicated migrator service)

Single-replica production deployment (no docker-compose, no autoscaling). With one server process there's no writer race and no crash-loop amplification, so the "dedicated migrator service gated by `service_completed_successfully`" pattern earns nothing here. Folding migrations into server boot removes an app, a Docker image, a CI matrix entry, and a compose service.

If this ever grows to N replicas, revisit: a one-shot migrator service becomes worth its weight again (single writer, fail-closed before any replica starts, slim app images).

## Manual escape hatch

For ad-hoc runs that shouldn't piggyback on a deploy (e.g. `CREATE INDEX CONCURRENTLY`, which can't run inside the migration transaction), run `bunx drizzle-kit migrate` from `packages/db` against the target `DATABASE_URL`.

## Forward-only (expand/contract)

1. **Expand** — additive migration. Ship.
2. **Migrate code** — read/write new shape. Ship.
3. **Contract** — drop old column/table. Ship.

Recovery from a bad schema change is a new forward migration.

## Running

- **Local / CI / Prod:** server applies pending migrations on boot before serving traffic. No separate step.

## Not in scope

Long-running data backfills needing domain code belong in BullMQ jobs, not the migration pipeline.
